## 3. Trust Service Criteria — Per-Change Review

### CC6.1 Logical Access

**Status: ✅ no regression**

- All new endpoints inherit existing middleware (rate limiting, security headers, NextAuth).
- The Falco webhook (`/webhooks/falco`) is unauthenticated *by user session* — it is authenticated by HMAC instead, same as the other webhook routes (host-agent, crowdsec, wazuh).
- The Trivy gateway tools execute in the gateway container, which itself enforces `JOIN_TOKEN`/`GATEWAY_AUDIT_SECRET` auth on its `/tools/execute` HTTP path (pre-existing).

### CC6.2 Authentication

**Status: ✅ no regression**

- Falco webhook: HMAC-SHA256 with 5-min replay window, identical to host-agent webhook (PR6).
  - Header: `X-Orion-Falco-Signature`
  - Secret: `FALCO_WEBHOOK_SECRET` env var, generated by `bootstrap.sh`, mirrored to Vault KV at `secret/orion/falco/webhook_secret`.
  - Loopback fallback only enabled when secret is unset (`warnMissingWebhookSecret` from `webhook-auth.ts`).
- NVD API: outbound only. Uses an `apiKey` request header. Stored in `SecurityConfig.NVD_API_KEY` + Vault KV.

### CC6.3 Role-Based Access

**Status: ✅ no regression**

- No new admin routes added.
- Existing per-handler `requireAdmin()` and gateway auth checks unchanged.

### CC6.5 Boundary Protection

**Status: ⚠️ deliberate exception, mitigated**

| Subject | Boundary | Mitigation |
|---|---|---|
| Falco container | Runs `privileged: true`, `pid: host` | Required by Falco's syscall-capture mechanism even in eBPF mode. Mitigated by: read-only host mounts (`/proc`, `/boot`, `/lib/modules`, `/usr`, `/etc` all `:ro`); pinned image version (`falcosecurity/falco-no-driver:0.38.2`); single-purpose container (no inbound user traffic). |
| Trivy outbound to public APIs | New egress: CISA, FIRST, NIST | All HTTPS; per-call 15–30s timeouts; no PII sent (CVE IDs only); rate-limited via process-local token bucket so we can't be ratelimit-banned. |
| `kubectl_get_events` | Calls `kubectl` with `--raw` URL path when resourceVersion provided | Input validated by `/^[0-9]+$/` regex before insertion. Namespace validated by `AGENT_RE` (`^[a-zA-Z0-9_-]+$`). |

### CC6.7 Data Injection

**Status: ✅ no regression**

| Code path | Defense |
|---|---|
| `kubectl_get_events` gateway tool | argv-array `execFile` (never shell); inputs gated by regex (`resourceVersion` digits-only, `namespace` AGENT_RE) |
| `trivy_scan_image` | argv-array `execFile`; image ref gated by `/^[A-Za-z0-9._/:@\-]+$/` — no whitespace, no shell metachars |
| `trivy_scan_k8s` | argv-array `execFile`; namespace gated by DNS-1123 regex |
| `trivy_scan_host` | No inputs |
| Falco webhook | All user-derived strings (rule, container.name, hostname) are serialized into the `rawEvent` JSONB column — never interpolated into SQL or shell. Zod schema validates wire shape. |
| K8s events normalizer | `metadata.uid` + `metadata.resourceVersion` taken verbatim; used only as opaque identifiers and as `rawEvent` payload. |

Argv-array construction prevents shell-level injection. Tight regex gates provide defense in depth at the input boundary. Note: the current `IMAGE_REF_RE` accepts a leading `-` (e.g. `-rm`) which Trivy's argv parser could mis-interpret as a flag — Trivy would error rather than execute anything harmful, but a leading-character guard is a worthwhile Phase 3.5 hardening (`/^[A-Za-z0-9]/` prefix check).

### CC6.6 Rate Limiting / System Failure

**Status: ✅ inherits middleware**

The Falco webhook is unauthenticated-by-session and Falcosidekick fires one POST per alert — without rate limiting, a misbehaving Falcosidekick could storm the route. The existing middleware (`apps/web/src/middleware.ts`, Redis-backed per-IP token bucket) applies to all `/api/*` paths and is not bypassed for `/webhooks/*`. The Falco webhook therefore inherits the same rate ceiling as the other webhooks (host-agent, crowdsec, wazuh). If a fleet of managed envs proves to exceed those limits at steady state, a dedicated higher per-IP ceiling for `/webhooks/falco` is a follow-up.

### CC6.8 Authorized Software

**Status: ✅ pinned images**

All new container images are pinned to specific tags:
- `falcosecurity/falco-no-driver:0.38.2`
- `falcosecurity/falcosidekick:2.30.0`
- Trivy is installed into the gateway image at build time (separate follow-up — currently `ENABLE_TRIVY=true` gates registration).

Helm chart references (`falcosecurity/falco` for managed K8s) are documented but not pinned in this repo — those are applied by the env-registration flow which should pin a known-good chart version.

### CC7.1 Monitoring / Audit

**Status: ✅ improved**

Phase 2 + 3 *add* monitoring surfaces; they don't reduce existing audit coverage:

- New SIEM sources: `falco`, `k8s_events`, `trivy`, `trivy_k8s`.
- New `VulnerabilityFinding` history (timestamped `firstSeenAt` / `lastSeenAt` / `fixedAt`).
- New `EnvironmentSourceHealth` heartbeat tracking → can detect a silenced data plane.
- `vuln.kev_critical` + `vuln.epss_storm` correlation rules give the Warden agent a signal to react on.

No new `AuditLog` write paths were introduced (the existing audit pipeline already covers admin actions; Phase 2/3 are purely ingest).

### C1 Encryption

**Status: ✅ no new plaintext-sensitive data**

| New stored field | Sensitivity | Treatment |
|---|---|---|
| `FALCO_WEBHOOK_SECRET` (env var, Vault, `.env`) | High | Same env-var pattern as the existing `HOST_AGENT_WEBHOOK_SECRET`; mirrored to Vault for rotation. |
| `NVD_API_KEY` (env var, Vault, `SecurityConfig`) | Medium | Vault is canonical; `SecurityConfig` row mirrored for app reads. Never logged. |
| `VulnerabilityFinding.rawTrivy` (JSONB) | Low | Trivy output is structured CVE data — no PII, no secrets. |
| `SecurityConfig.KEV_CACHE` (JSONB) | Public | CISA's published catalog. |
| `EnvironmentSourceHealth.lastWatermark` | Low | K8s `resourceVersion` (opaque string). |

No new fields hold PII. No new fields require encryption at rest beyond what Vault provides for the secret-class entries.

### C1 Input Validation

**Status: ✅ all new boundaries validated**

| Boundary | Validation |
|---|---|
| Falco webhook body | `falcoAlertSchema` (Zod) requires `rule` + `priority`; preserves `output_fields` verbatim |
| Falco `environmentId` from `output_fields` | Type-checked (`typeof === 'string' && length > 0`); referential integrity checked against `Environment` table (except literal `"host"`) |
| K8s event poller `resourceVersion` (DB read) | Validated by gateway tool regex before use |
| Trivy image ref / namespace | Tight regexes (see CC6.7) |
| CVE enrichment inputs | URL-encoded; size-capped batches (100 CVEs/EPSS request) |

### C1 Data Protection

**Status: ✅ no regression**

- New rows scoped by `environmentId` with `ON DELETE CASCADE` — managed-env data dies with the env.
- `VulnerabilityFinding` `@@unique([environmentId, target, cveId, packageName])` prevents cross-env data bleed in upsert paths.

### C6 Availability

**Status: ✅ defensive**

- Trivy gateway tools: in-process semaphore caps to 1 concurrent scan per gateway (prevents OOM).
- Output capped at 8 MiB (prevents response-stream blowup).
- Per-tool timeouts (image 120s, k8s 180s, host 300s).
- K8s poller: per-env error isolation — one failing env doesn't block others in the cycle.
- CVE enrichment: best-effort (returns empty / null on failure) — never blocks a scan.
- KEV cache 24h TTL with stale-cache fallback on fetch failure.
- NVD token bucket: never exceeds quota.
- Daily scan guard: date-keyed so it runs exactly once per day even across worker restarts.

## 4. Flagged Deliberate Choices

These are call-outs that an auditor will want to discuss. Each one is the *correct* trade-off for this stack but is worth surfacing.

### 4.1 Falco runs privileged on every host

Required by Falco's syscall capture even in eBPF mode. Mitigated by read-only host mounts and version pinning. Alternative — Falco modern-eBPF without privileged — is not yet stable in the 0.38 line; revisit when 0.40+ is GA.

### 4.2 Single shared `FALCO_WEBHOOK_SECRET` across the fleet

Every Falcosidekick instance (Orion host + every managed env) signs with the same secret. The plan calls for per-env secrets via Vault rotation; for now we use one shared secret to ship Phase 2 faster. Per-env rotation is tracked as a Phase 2.5 follow-up: route would look up `secret/orion/<env>/falco/webhook_secret` from `SecurityConfig` based on `output_fields.environmentId`.

### 4.3 NVD enrichment is best-effort

If NVD is down or rate-limit-exhausted, findings ship without `attackVector` / `cvssVector`. Severity falls back to CVSS-score-times-10 + KEV/EPSS modifiers. We prefer "ship a finding with partial enrichment" over "block scans on enrichment availability."
